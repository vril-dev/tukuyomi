# tukuyomi v1.2.0 Release Notes

Status: draft
Baseline: v1.1.8

v1.2.0 introduces Tukuyomi Center, IoT / Edge enrollment, and the first
operator-facing device approval workflow. It also tightens install roles and
frontend build hygiene.

## Highlights

- Added Center mode as part of the same single binary. A host can now be
  installed as either a Gateway or Center by selecting `INSTALL_ROLE`.
- Added a dedicated Center UI with login, status, user account management, and
  device enrollment approval screens.
- Added IoT / Edge mode on the Gateway. When enabled, public proxy traffic is
  locked until the local Gateway identity is approved by Center.
- Added one-time enrollment tokens in Center. Gateway generates an Ed25519
  device identity, sends a signed enrollment request, and stores Center approval
  state locally.
- Added registered-device lifecycle controls in Center, including approval
  revoke and archive for revoked devices.
- Added Gateway-side re-approval handling so a revoked device can request
  approval again with a new token while keeping the same device identity.
- Added `make center-preview-up`, `make gateway-preview-up`, and
  `make fleet-preview-up` preview flows for local Gateway + Center testing.
- Frontend dependency locks were refreshed, npm audit findings were cleared,
  and the Gateway UI was code-split by page to remove the Vite chunk-size
  warning.

## Operator Notes

- IoT / Edge mode is off by default. Web/VPS deployments should leave it off
  unless this Gateway must be enrolled and controlled as an edge device.
- In IoT / Edge mode, the Gateway public proxy path rejects traffic while the
  device is not approved. The admin UI/API remains reachable for local recovery.
- Center approval is not a push channel yet. Gateway refreshes Center device
  state by explicit UI action and the configured polling interval.
- Enrollment tokens are one-time secrets. Center shows the token only at
  creation time. If it is lost, create a new token instead of trying to recover
  the old one.
- Gateway stores the generated device private key locally. Center stores the
  public key fingerprint and approval state.
- The public key fingerprint shown in Gateway and Center is SHA-256 over the
  DER-encoded Ed25519 public key, rendered as lowercase hex.
- Gateway rejects duplicate enrollment requests while local state is `pending`
  or `approved` so replacement tokens are not accidentally consumed.
- Gateway allows re-approval only from replacement states such as `revoked`,
  `archived`, `failed`, or `product_changed`.
- Revoking an enrollment token also revokes devices enrolled through that token,
  except devices already archived for audit retention.
- Archiving hides revoked devices from the default registered-device list
  without deleting their audit trail.
- `tukuyomi center` is a separate process mode, but it is built from the same
  binary as the Gateway.

## Deployment

- `make install TARGET=linux-systemd INSTALL_ROLE=gateway` installs a Gateway.
- `make install TARGET=linux-systemd INSTALL_ROLE=center` installs Center.
- `INSTALL_ROLE` defaults to `gateway` for compatibility with existing install
  commands.
- The installer renders role-specific systemd units and role-specific runtime
  environment files.
- The preview flow was split into Gateway, Center, and fleet targets:
  - `make gateway-preview-up`
  - `make center-preview-up`
  - `make fleet-preview-up`
- `GATEWAY_PREVIEW_PERSIST=1` keeps Gateway preview DB/config state across
  preview restarts.
- `CENTER_PREVIEW_PERSIST=1` keeps Center preview DB state across preview
  restarts.

## Admin UI

- Gateway Options now includes IoT / Edge mode controls, Center enrollment
  status, Center URL, enrollment token entry, and Center status refresh.
- Gateway Options exposes the Center polling interval for IoT / Edge status
  refresh.
- Gateway shows whether proxy traffic is available or locked by device approval
  state.
- Center UI follows the same visual style as the Gateway admin UI.
- Center Status focuses on device overview counts.
- Center Device Approvals manages enrollment tokens, pending approvals,
  registered devices, revoked devices, and archived devices.
- Center User allows username, email, password, and personal-access-token
  management.
- Gateway and Center browser sessions are separated so both UIs can be used in
  the same browser at the same time.

## Build and Development

- Gateway and Center UI builds now declare `node >=22.12.0` and `npm >=10`.
- `.nvmrc` now points to Node 22.
- CI already uses Node 22, and the sample deployment Dockerfile builds UI with
  Node 22.
- Gateway UI routes are lazy-loaded so large admin pages do not inflate the
  initial JavaScript bundle.
- `npm audit --audit-level=moderate` reports zero findings for both Gateway and
  Center UI packages after the dependency lock refresh.

## Documentation

- Added device enrollment operation docs for Center token creation, Gateway
  approval request, Center approval, status refresh, and re-approval.
- Updated the root README with IoT / Edge enrollment and Center install flow.
- Updated binary deployment docs for `INSTALL_ROLE=gateway|center`.
- Added Center service/env deployment examples.
- Removed old product-family comparison pages. Tukuyomi is documented as one
  product with Gateway, Center, Web, and IoT capabilities instead of separate
  product names.

## Compatibility and Migration

- DB migrations are required before running v1.2.0.
- New DB schema covers Center enrollment tokens, Gateway edge device identity,
  Center status cache, device revocation, and device archive state.
- Existing Gateway installs remain Gateway installs because `INSTALL_ROLE`
  defaults to `gateway`.
- Existing non-IoT traffic behavior is unchanged while `edge.enabled` is false.
- `edge.enabled` is explicitly present in bundled config examples and defaults
  to false.
- Building the UI with Node 18 is no longer supported. Use Node 22 as declared
  by `.nvmrc` and package `engines`.

## Known Limits

- Center currently handles device enrollment and approval. It does not yet push
  Gateway configuration, collect logs, or manage Gateway binary upgrades.
- Center state refresh is polling-based. Immediate push from Center to Gateway
  is not implemented in this release.
- Device approval protects the Gateway proxy path only when IoT / Edge mode is
  enabled.
