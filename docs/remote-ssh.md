# Remote SSH

Remote SSH lets an operator open a short-lived maintenance session to an
approved Gateway without exposing an inbound Gateway port.

Two operator paths are supported:

- **Web Terminal** from Center UI. This is the normal interactive path for
  operators who already work in Center.
- **CLI handoff** with `tukuyomi remote-ssh`. This remains available for
  emergency, automation, and local SSH workflows.

Both paths use the same Center session model, Gateway outbound stream, device
policy, TTL, and audit trail.

## Security Model

- Remote SSH is disabled by default on both Center and Gateway.
- Center requires an approved device, an enabled per-device policy, a reason,
  TTL, and an authenticated admin session or token.
- Gateway accepts pending sessions only when they are signed by the pinned
  Center Ed25519 signing key.
- Gateway connects outbound to Center and runs an embedded SSH server inside
  the `tukuyomi` process.
- Web Terminal uses a Center-owned one-time Ed25519 operator key and relays PTY
  bytes through a browser WebSocket.
- CLI access uses a one-time Ed25519 public key generated by
  `tukuyomi remote-ssh`.
- The CLI waits for the Gateway SSH host public key through Center and writes a
  temporary pinned `known_hosts` file before printing the SSH command.
- Center and CLI require HTTPS for Remote SSH by default. HTTP is available
  only for explicit local testing flags.
- Gateway refuses to run a shell as root unless
  `remote_ssh.gateway.embedded_server.run_as_user` is set.
- Gateway starts shells with a minimal environment instead of inheriting the
  `tukuyomi` process environment.
- Port forwarding, SFTP, SCP, agent forwarding, and arbitrary SSH subsystems
  are not enabled.

## Configuration

Center side:

```json
{
  "edge": {
    "enabled": true
  },
  "remote_ssh": {
    "center": {
      "enabled": true,
      "max_ttl_sec": 900,
      "idle_timeout_sec": 300,
      "max_sessions_total": 16,
      "max_sessions_per_device": 1
    }
  }
}
```

Gateway side:

First fetch the Center signing public key from an authenticated Center admin
session:

```sh
curl "$CENTER/center-api/remote-ssh/signing-key" \
  -H "Authorization: Bearer $TUKUYOMI_ADMIN_TOKEN"
```

Set the returned `public_key` as
`remote_ssh.gateway.center_signing_public_key` on each Gateway:

```json
{
  "edge": {
    "enabled": true
  },
  "remote_ssh": {
    "gateway": {
      "enabled": true,
      "center_signing_public_key": "ed25519:REPLACE_WITH_CENTER_PUBLIC_KEY",
      "center_tls_ca_bundle_file": "conf/center-ca.pem",
      "center_tls_server_name": "center.example.local",
      "embedded_server": {
        "enabled": true,
        "shell": "/bin/sh",
        "working_dir": "/",
        "run_as_user": "tukuyomi"
      }
    }
  }
}
```

The Gateway embedded server will not accept Remote SSH sessions without this
pinned key.

`center_tls_ca_bundle_file` is optional when Center uses a public CA or ACME
certificate. For closed-network or self-signed Center certificates, put the
Center CA certificate in this PEM bundle and distribute it to each Gateway.
`center_tls_server_name` is optional, but useful when Gateway reaches Center by
IP while the certificate is issued to a DNS name.

When bootstrapping a Center-protected Gateway, the same trust settings can be
written into the Gateway config:

```sh
tukuyomi bootstrap-center-protected-gateway \
  --center-url "https://center.example.local" \
  --center-ca-bundle "conf/center-ca.pem" \
  --center-server-name "center.example.local"
```

## Center UI Operation

![Center Remote SSH screen](images/ui-samples/28-center-remote-ssh.png)

1. Open `Device Approvals`.
2. Select a registered Gateway and open `Manage`.
3. Open `Remote SSH` in the selected device menu.
4. Confirm that `Center service` is enabled.
5. Enable the per-device policy, set `Max TTL seconds`, and save.
6. Enter a reason and TTL under `Web terminal`.
7. Adjust `Scrollback rows` if a smaller or larger browser-side terminal
   history is needed.
8. Click `Open terminal`.

Center creates a pending Remote SSH session. The browser WebSocket opens
immediately, then waits for the Gateway to pick up the pending session during
the next signed Center status poll. With a 30 second poll interval, the
operator may wait up to roughly one polling interval plus SSH setup time.

When the Gateway attaches, the terminal shows `Terminal connected.` and the PTY
is interactive.

Use `Close terminal` to close the current Web Terminal. The Sessions table also
shows recent sessions. Active or pending sessions can be terminated from the
status cell.

`Scrollback rows` controls how many old terminal rows the browser keeps. It
does not write a server-side recording. Session recording is a separate audit
feature and is not enabled by this setting.

## CLI Operation

Enable the per-device policy from the Center API:

```sh
curl -X PUT "$CENTER/center-api/devices/$DEVICE/remote-ssh/policy" \
  -H "Authorization: Bearer $TUKUYOMI_ADMIN_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"enabled":true,"max_ttl_sec":900,"require_reason":true,"allowed_run_as_user":"tukuyomi"}'
```

Start a local operator tunnel:

```sh
export TUKUYOMI_ADMIN_TOKEN="$TOKEN"
tukuyomi remote-ssh \
  --center "https://center.example.com" \
  --center-ca-bundle "conf/center-ca.pem" \
  --center-server-name "center.example.local" \
  --device "$DEVICE" \
  --reason "maintenance"
```

The command prints the exact local `ssh` command. Keep
`tukuyomi remote-ssh` running while the SSH session is active.

Omit `--center-ca-bundle` and `--center-server-name` when Center uses a public
CA certificate and the URL hostname matches the certificate.

For local-only testing with an HTTP Center URL, add `--allow-insecure-http` or
set `TUKUYOMI_REMOTE_SSH_ALLOW_INSECURE_HTTP=1`. Do not use this for production
Gateways.

## Operational Notes

- Remote SSH pickup depends on the Gateway-to-Center status polling interval.
  A short delay after `Open terminal` is normal. A wait far beyond the polling
  interval should be treated as an operational fault.
- TTL closes the session even if the terminal is still open.
- Idle timeout closes sessions that stop sending traffic.
- Terminating a session from Center closes any active Web Terminal or CLI
  relay pair and releases the per-device session limit.
- Center signing key rotation requires Gateway trust refresh. Approved
  Center-managed Gateways can refresh the key from the approved Center.
